Webserv Request Lifecycle


Step 1: The User Opens the Browser
A user sits at their computer and types http://localhost:8080/login.php?redirect=dashboard into their browser's address bar, then presses Enter. The browser immediately begins constructing an HTTP request. It knows the protocol is HTTP, the host is localhost, the port is 8080, the path is /login.php, and the query string is ?redirect=dashboard. The browser also gathers other information it has stored, such as any cookies previously set by this domain. In this case, the user has never visited before, so no cookies exist yet. The browser prepares a TCP socket using socket(AF_INET, SOCK_STREAM, 0) and attempts to connect to 127.0.0.1:8080 through a three-way handshake: SYN, SYN-ACK, ACK. Once the connection is established, raw bytes begin flowing from the browser toward your server.


Step 2: The Server Listens
Your server has already been running. During its startup phase, which corresponds to Phase 2 and Phase 4 of your project, the back-end peer parsed the configuration file and discovered that the server should listen on 127.0.0.1:8080. The back-end created a listening socket, set it to non-blocking mode with fcntl(fd, F_SETFL, O_NONBLOCK), bound it to the address, and called listen(). This socket, let's say it has file descriptor 3, was added to the single poll() array with the POLLIN event flag. The server is now asleep inside poll(), waiting for something to happen. It is not burning CPU cycles checking repeatedly. It is completely idle until the kernel wakes it up.


Step 3: A Connection Arrives
The kernel network stack receives the incoming SYN packet from the browser. It completes the handshake and creates a new connected socket on the server side. Because the listening socket fd=3 now has a pending connection, poll() returns and reports that fd=3 is readable. The back-end's accept_new_client() function runs. It calls accept(3, &client_addr, &len) which returns a new file descriptor, let's say fd=5. The back-end immediately creates a Client object. This object stores fd=5, initializes read_buffer and write_buffer as empty strings, sets state to READING, sets server_idx to 0 pointing to the first server block in the config, and sets keep_alive to true. The new client socket fd=5 is added to the poll() array with POLLIN. The server loops back to poll() and goes back to sleep, now monitoring both the listener fd=3 and the new client fd=5.


Step 4: Reading the Request
The browser now sends the actual HTTP request as raw bytes over the TCP connection. The kernel buffers these bytes and delivers them to the server's socket. poll() wakes up again, this time reporting that fd=5 has POLLIN. The back-end calls read_from_client(fd=5), which executes recv(5, buf, 4096, 0). Because the socket is non-blocking, recv() returns immediately. If only partial data has arrived, it returns whatever is available and the back-end appends it to client->read_buffer. This process repeats across multiple poll() cycles. The back-end keeps appending bytes until it detects the sequence \r\n\r\n, which marks the end of the HTTP headers. For a GET request, there is no body, so the request is complete as soon as the headers end. At this moment, the back-end triggers the handoff to the front-end by calling FrontEnd::handleRequest(*client, config).


Step 5: Parsing HTTP
The front-end peer now takes control. It receives the raw request string from client->read_buffer. The first task is parsing the request line. It splits "GET /login.php?redirect=dashboard HTTP/1.1" into three parts: the method "GET", the URI "/login.php", and the query string "redirect=dashboard", plus the HTTP version "HTTP/1.1". Next, the front-end parses all headers into a std::map<std::string, std::string>. It extracts Host, User-Agent, Accept, and Connection. Because this is the first visit, the Cookie header is absent. The front-end creates an HttpRequest object and populates it with all these parsed fields. It stores this object by casting it to void* and assigning it to client->request_obj. The request is now a structured C++ object instead of raw bytes.


Step 6: Routing the Request
With the parsed request in hand, the front-end enters Phase 7, routing. It looks up client->server_idx to determine which ServerConfig applies, in this case the first server block listening on port 8080. It then iterates through all Location blocks in that server config, looking for the longest prefix match against the URI "/login.php". Suppose the config has a location / with cgi_extension .php and cgi_path /usr/bin/php-cgi. The URI "/login.php" matches this location. Because the file extension is .php and matches the configured cgi_extension, the front-end marks this as a CGI request. It builds a RouteResult structure containing the matched location, the resolved file path "./www/login.php", the interpreter "/usr/bin/php-cgi", and sets isCgi to true. The front-end also verifies that "GET" is in the allow_methods list for this location.


Step 7: Preparing CGI Execution
Because route.isCgi is true, the front-end skips static file serving entirely and moves to Phase 11, CGI. The front-end builds a map of environment variables that the CGI script will need. It sets REQUEST_METHOD to "GET", QUERY_STRING to "redirect=dashboard", CONTENT_LENGTH to "0", SCRIPT_NAME to "/login.php", PATH_INFO to "/login.php", and PATH_TRANSLATED to the absolute filesystem path. It also copies relevant HTTP headers into the environment, such as HTTP_HOST and HTTP_USER_AGENT. Since no cookies exist yet, HTTP_COOKIE is left empty. The front-end then calls BackEnd::startCgi(client, "/usr/bin/php-cgi", "./www/login.php", env_map), handing control back to the back-end to actually spawn the process.


Step 8: Forking the CGI Process
The back-end receives the call to startCgi(). It creates two pipes: one for standard input and one for standard output. Both pipe ends are set to non-blocking. The back-end adds the read end of the stdout pipe to the poll() array with POLLIN, so it will be notified when the CGI produces output. Then it calls fork(). In the child process, the back-end closes unused pipe ends, duplicates the stdin pipe read end to file descriptor 0 and the stdout pipe write end to file descriptor 1 using dup2(), changes into the script's directory with chdir(), sets all environment variables with setenv(), and finally calls execve("/usr/bin/php-cgi", args, environ). If execve succeeds, the child process is now the PHP interpreter running login.php. If it fails, the child exits with code 1. The parent process stores the child's PID and the pipe file descriptors in the Client object, sets client->state to CGI_RUNNING, and returns to the poll() loop.


Step 9: The CGI Script Runs
The PHP interpreter begins executing login.php. It reads the environment variables to understand the request. It sees REQUEST_METHOD=GET and QUERY_STRING=redirect=dashboard. It attempts to start a PHP session with session_start(), but since no session cookie was passed, it creates a new one internally. The script checks if a user is logged in. They are not, so it outputs an HTML login form instead of redirecting. The script writes to its standard output, which due to dup2() is actually the write end of the pipe back to the server. The output looks like "Content-type: text/html\r\n\r\n<html><body><form action='/login.php' method='POST'>...". Once the script finishes, it exits, closing its stdout pipe.


Step 10: Reading CGI Output
Back in the parent server process, poll() eventually reports that the stdout pipe has POLLIN data available, or the pipe closes. The back-end reads from the pipe into client->cgi_output_buffer using repeated read() calls in a loop. Because the pipe is non-blocking, it handles EAGAIN gracefully. When read() returns 0, indicating EOF, the back-end knows the CGI has finished writing. It then calls waitpid(client->cgi_pid, &status, WNOHANG) to reap the child process. If the child has exited, the back-end closes both pipe ends, removes the pipe file descriptors from the poll() array, and calls FrontEnd::handleCgiComplete(client, config, exit_status) to hand the raw CGI output back to the front-end for formatting.


Step 11: Formatting the Response
The front-end receives the raw CGI output from client->cgi_output_buffer. It splits the output at the first occurrence of \r\n\r\n, separating CGI headers from the CGI body. In this case, the CGI header is "Content-type: text/html" and the body is the HTML form. The front-end now builds a proper HttpResponse object. It sets the HTTP status code to 200, copies the Content-type from the CGI headers, calculates the Content-Length from the body size, and sets Connection: keep-alive based on the original request's HTTP version. Because this is the first request and no valid session cookie was found, the front-end also interacts with the SessionManager. It creates a new session, generating a random 16-character hex ID like "b8e3f1a2c5d9". It stores some default data in the session, such as "theme" = "light". It then adds a Set-Cookie header to the response: "Set-Cookie: session_id=b8e3f1a2c5d9; Path=/". The front-end serializes the entire response into a single string and stores it in client->write_buffer. It sets client->keep_alive = true, sets client->state = WRITING, and calls BackEnd::setClientWritable(client).


Step 12: Sending the Response
The back-end receives the call to setClientWritable(). It modifies the client's entry in the poll() array to include POLLOUT in addition to POLLIN. On the next poll() iteration, the kernel reports that fd=5 is writable. The back-end calls write_to_client(fd=5), which executes send(5, client->write_buffer.c_str(), client->write_buffer.size(), 0). Because the socket is non-blocking, send() returns immediately with however many bytes the kernel's outgoing buffer can accept. If this is less than the full response, the back-end erases the sent portion from write_buffer and keeps POLLOUT active. On subsequent poll() cycles, it sends the remainder. Once the entire buffer is sent and write_buffer is empty, the back-end removes POLLOUT from the client's poll events, leaving only POLLIN.


Step 13: Connection Management
With the response fully transmitted, the back-end checks client->keep_alive, which the front-end set to true. Because HTTP/1.1 defaults to persistent connections, the back-end resets the client state. It sets client->state = READING, clears client->read_buffer, updates client->last_activity to the current time, and keeps fd=5 in the poll() array with POLLIN. The connection remains open, waiting for the browser to send another request. If the browser had sent Connection: close, or if this were HTTP/1.0 without Connection: keep-alive, the back-end would instead call close(fd=5), remove the client from the poll set, and delete the Client object.


Step 14: The Browser Renders
The browser receives the complete HTTP response from the server. It parses the status line HTTP/1.1 200 OK, confirming success. It reads the Content-Type: text/html header and knows to render the body as a web page. Crucially, it sees the Set-Cookie: session_id=b8e3f1a2c5d9; Path=/ header and stores this cookie in its cookie jar. The cookie will be sent back to localhost:8080 on every subsequent request. The browser then parses the HTML body, builds a DOM tree, and renders the login form on screen. The user sees two input fields, one for username and one for password, plus a login button. The TCP connection remains open in the background, ready for the next request.


Step 15: The Second Request with Session
The user fills in the form and clicks Login. The browser constructs a new HTTP request, this time a POST to /login.php. Because the browser now has a cookie stored for this domain, it automatically includes it in the request headers: Cookie: session_id=b8e3f1a2c5d9. This request travels through the same TCP connection, hits the server's poll() loop, gets accepted on the same fd=5, and is read into client->read_buffer. The front-end parses it in Phase 6, and this time the Cookie header is present. The front-end extracts session_id=b8e3f1a2c5d9 and passes it to SessionManager::getSession(). The session is found, and the front-end can now personalize the response or validate the login credentials against stored session data. The cycle repeats: routing, CGI or static file, response building, and sending back to the browser.









Summary of the Complete Flow
The browser initiates a TCP connection to the server's listening socket. The server's poll() loop detects the new connection, accepts it, and begins reading raw HTTP bytes into a buffer. Once the request is complete, the back-end hands it to the front-end. The front-end parses the HTTP request, routes it to the correct location, and determines whether to serve a static file or execute a CGI script. For CGI, the front-end prepares environment variables and asks the back-end to fork and execute the interpreter. The back-end manages the child process and its pipes, reading the CGI output when ready. The front-end then formats that output into a proper HTTP response, optionally creating or reading session cookies, and stores the response in the client's write buffer. The back-end sends this response through the non-blocking socket, manages partial writes across multiple poll() cycles, and finally resets the connection for keep-alive or closes it. The browser receives the response, stores any cookies, and renders the page. Every subsequent interaction follows this same pipeline, with the session cookie allowing the server to recognize returning users.
